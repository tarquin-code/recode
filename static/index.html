Quick Start

Recode converts your video library to HEVC (H.265), saving significant disk space while maintaining quality. Here's how to get started:

Workflow
1. Select Library
→
2. Scan Files
→
3. Select & Queue
→
4. Encode
→
5. Done!
  1. Select a library — Choose a Plex library from the dropdown, or select "Custom folder" and browse to any directory.
  2. Scan — Click Scan to discover video files. Results are cached, so subsequent scans are near-instant.
  3. Configure settings — Choose a preset or set custom CQ/bitrate/speed in the left sidebar. Adjust audio and filter options as needed.
  4. Queue files — Check the files you want to encode, then click "Add Selected to Queue".
  5. Monitor — Switch to the Encoding tab to watch real-time progress, or let it run in the background.

Encoder: GPU vs CPU

GPU (hevc_nvenc)
  • NVIDIA CUDA hardware acceleration
  • 10-50x faster than realtime
  • Recommended for most users
  • Multi-GPU support with auto load balancing
  • Concurrent encodes auto-limited by VRAM
CPU (libx265)
  • Software encoding
  • Much slower but 10-20% smaller files
  • Better for archival quality
  • Use when GPU isn't available
  • Thread count configurable in Settings → General

Content Filters

  • Skip 4K files — Don't encode 4K/UHD content. Useful if you only want to target lower-res files.
  • HDR only — Only encode HDR/DV content, skip SDR files.
  • Dolby Vision mode — See the Dolby Vision tab for full details on DV P5/P7/P8 handling and conversion options.

Output Control

  • Discard if larger — Delete output if it's bigger than original. Common with already-efficient H.264 encodes.
  • Delete original — Remove source file after successful encode.
Warning: "Delete original" permanently destroys the source file. This cannot be undone. Use with caution.

Pause / Resume / Cancel

Active encodes have Pause and Cancel buttons in the header bar. Pausing freezes ffmpeg with zero resource usage. Resume continues exactly where it left off. Cancel terminates the encode and cleans up temporary files.

Permission Checks

  • Before each encode, write permissions are verified for the temp directory and output directory
  • If permissions are insufficient, a red warning banner is shown after scanning
  • The encode fails immediately with a clear error rather than after a long encode

Default vs Per-Stream

The sidebar sets the default audio codec and bitrate. In the scan results, expand any file to override per-stream:

  • Default — Uses the sidebar setting for this stream
  • Copy (passthrough) — Keep original audio untouched
  • Specific codec — Override with Opus, AAC, AC3, or EAC3

When set to "Default", the bitrate selector is disabled since the sidebar default applies.

English Only

When enabled, non-English audio tracks are automatically excluded. This can significantly reduce file size for media with many language tracks.

Keep Only Audio Type

Filters which audio streams are included by codec type. For example, set to "EAC3" to only keep EAC3 tracks and drop TrueHD, AAC, etc.

Dolby Vision + Audio: When encoding DV content, use Passthrough for audio. Re-encoding audio (even to EAC3) can prevent devices like Fire Stick from triggering the DV playback pipeline.

Dolby Vision

DV Mode Options

Skip (default)
DV files are not encoded. They remain untouched in the queue.
Convert to HDR10
Strips DV metadata, keeps the HDR10 base layer. Reduces complexity, widest compatibility.
Convert to DV P8.4
Re-encodes video and converts DV metadata to Profile 8.4. Also upgrades HDR10 content to DV P8.4.

DV P5 → P8.4 Conversion Pipeline

When converting DV Profile 5 content, the encode uses a 4-step post-processing pipeline:

Conversion steps
1. Extract RPU
→
2. Extract HEVC
→
3. Inject RPU
→
4. Mux with mkvmerge
  • Step 1: Extracts the Dolby Vision RPU (Reference Processing Unit) from the source and converts it using dovi_tool
  • Step 2: Extracts the raw HEVC bitstream from the encoded output
  • Step 3: Injects the converted RPU into the encoded HEVC stream
  • Step 4: Muxes the DV-injected video with audio/subtitles using mkvmerge, then patches the container with correct P8.4 compatibility signaling

DV Profile 5 (Streaming)

P5 uses IPTPQc2 color space (single layer). During encoding, libplacebo (Vulkan) converts colors from IPTPQc2 to BT.2020/PQ, and the RPU is converted to Profile 8.4 format.

DV P5 encodes use Vulkan/libplacebo for color conversion, which requires GPU access. The Vulkan device is automatically matched to the NVENC GPU assignment.

HDR10 → DV P8.4 Upgrade

HDR10 content can be upgraded to DV P8.4 by generating a new RPU from the HDR10 metadata (MaxCLL, MaxFALL, mastering luminance). The result has DV dynamic metadata wrapping the HDR10 base layer.

Playback Compatibility

Important: For DV playback on Fire Stick and similar devices:
  • Audio must be passthrough (copy) — re-encoded audio breaks the DV hardware playback pipeline
  • The file uses MKV container with dvvC configuration record
  • Profile 8.4 (compatibility_id=4) provides broad device support

DV Without GPU (CPU Fallback)

  • DV P5 → HDR10 conversion works WITHOUT libplacebo — strips DV metadata and sets HDR10 color flags
  • DV P5 → P8.4 conversion REQUIRES libplacebo (Vulkan + GPU)
  • The "Convert to DV P8.4" option is automatically disabled in the UI when libplacebo is not available
  • If no GPU is available, use "Convert to HDR10" mode for DV files

Output Filename

Encoded files follow the pattern: Movie_h265_DV-P8_suffix.mkv

The suffix is configurable in Settings → General → Encode File Suffix (default: recode).

You can manually select a GPU from the sidebar dropdown, or leave it on "Auto" for automatic load balancing.

CPU-Only Mode

  • When no GPU is detected, the encoder selector is forced to CPU and the GPU dropdown is hidden
  • CPU jobs work without any GPU — no GPU assignment needed
  • CPU encoding uses libx265 which is slower but produces slightly smaller files

GPU OOM Handling

If an encode runs out of GPU memory, it's automatically re-queued at the back of the queue (up to 3 retries). The job will try again when GPU memory is available.

Changing Concurrency

If you lower the Concurrent Encodes setting while jobs are running, excess encodes are immediately stopped and re-queued at the front of the queue. The oldest running jobs are kept; the newest are stopped first.

System Monitoring

The header bar shows live CPU/RAM/GPU utilization. The Encoding tab displays per-GPU graphs with:

  • Green line — GPU utilization %
  • Blue line — VRAM usage %
  • Orange line — Temperature (when available)

Each active encode card shows which GPU it's assigned to (e.g. "GPU 0", "GPU 1").

GPU stats poll every 2 seconds during encoding, every 5 seconds when idle. Polling pauses entirely when the browser tab is hidden.

Remote GPU Encoding

Recode uses RRP (Recode Remote Protocol) for offloading encodes to remote GPU servers. GPU servers connect OUT to your machine — no port forwarding needed on the GPU side. Everything runs over TCP with HMAC-SHA256 authentication.

How remote encoding works (reverse-connect)
GPU server connects to client
→
Auth + heartbeats
→
Job assigned via control channel
→
Data connection: FUSE mount + encode
→
Output returned (SHA256 verified)

Two connection types: a persistent control channel (heartbeats, job assignment, status) and a per-job data channel (FUSE file reads, progress, output transfer). The GPU server mounts the input file on-demand via FUSE — encoding starts immediately without uploading the entire file.

  • NAT-friendly — GPU servers connect out, no port forwarding needed on the GPU side
  • CUDA hardware decode on the remote GPU
  • Live progress with estimated speed and bitrate
  • Cancel propagation — cancelling a job kills ffmpeg on the remote server and cleans up
  • Automatic FUSE unmount and temp file cleanup on cancel, failure, or disconnect
  • Load balancing across multiple GPU servers with per-GPU max_jobs enforcement
  • Hardware encoder probing — tests actual GPU capabilities (e.g. auto-strips temporal-aq on Maxwell GPUs)
  • H.264 and H.265 capability badges for each connected GPU

Remote Server (Client Machine)

Enable Remote Server in Settings → Remote GPU. Set a port (default 9879) and shared secret. GPU servers connect to this port. Connected GPUs appear with green indicators showing name, IP, encoder capabilities, and job counts.

The header shows a "Server" indicator with a badge count of connected GPUs. The copyable address line makes it easy to configure GPU servers.

Remote Clients (GPU Server)

On the GPU server, add entries under Remote Clients in Settings → Remote GPU. Enter the client's address (hostname:port) and the shared secret. The GPU server connects out and registers its capabilities automatically. Auto-reconnects with exponential backoff if the connection drops.

The header shows a "Client" indicator when connected to a client. Connection errors are shown in tooltips on the status buttons.

Setup Steps

  1. On the client (machine with files): enable Remote Server in Settings → Remote GPU, set port and secret
  2. On each GPU server: install Recode, add the client under Remote Clients with address and secret
  3. The GPU server connects automatically — you'll see it appear in the client's connected GPUs list
  4. Select "Remote" or "Auto" in the GPU Target dropdown, or pick a specific GPU by name

Firewall Configuration

The client machine needs port 9879 (Remote Server) open for incoming GPU server connections. The web UI uses port 9877. GPU servers need no inbound ports — they connect out.

# On the CLIENT machine — open port 9879 for GPU server connections
# firewalld (RHEL/Fedora/Rocky/Alma)
firewall-cmd --permanent --add-port=9879/tcp
firewall-cmd --reload

# ufw (Ubuntu/Debian)
ufw allow 9879/tcp

# iptables
iptables -A INPUT -p tcp --dport 9879 -j ACCEPT
If you change the Remote Server port in Settings, update the firewall rule accordingly. GPU servers need no inbound ports open — they connect out to the client.

FUSE Requirements

The GPU server needs FUSE support for mount mode. The installer handles this automatically, but if setting up manually:

# Install fuse3
dnf install fuse3           # RHEL/Fedora
apt install fuse3           # Debian/Ubuntu
pacman -S fuse3             # Arch

# Enable user_allow_other in /etc/fuse.conf
echo 'user_allow_other' >> /etc/fuse.conf

Remote Server Status

  • • Green — Enabled and online (authenticated successfully)
  • • Orange — Enabled but offline or unreachable
  • • Red — Disabled

Status is checked automatically when settings are saved. The power button toggles enable/disable per server.

macOS GPU Server

Apple Silicon Macs can serve as remote GPU servers using VideoToolbox hardware encoding. The Mac connects out to your client machine — no port forwarding needed.

# Install macFUSE
brew install macfuse

# Copy binaries from the macos/ folder in the release
mkdir -p ~/recode && cp macos/* ~/recode/ && chmod +x ~/recode/*

# Connect to your client machine (reverse-connect)
~/recode/recode-remote connect --address client.example.com:9879 --secret xxx --name "Mac Studio" --ffmpeg ~/recode/ffmpeg
  • Requires macOS 11+ on Apple Silicon (M1/M2/M3/M4)
  • Requires macFUSE — approve kernel extension in System Settings → Privacy & Security
  • Encoder type (videotoolbox) auto-detected via ping
  • HDR10 passthrough supported

Auto-Detect Encoder

Remote server capabilities are detected automatically via ping. No manual configuration needed:

  • NVENC — NVIDIA GPU (Linux)
  • VideoToolbox — Apple Silicon (macOS)
  • QSV — Intel Quick Sync Video
  • AMF — AMD AMF
  • CPU — libx265 software encoding

RRP Binary

All remote functionality is in a single binary: recode-remote

recode-remote server --port 9878 --secret xxx --ffmpeg /path/to/ffmpeg
recode-remote client [ffmpeg args]
recode-remote ping     # Returns: OK encoders os/arch fuse=true

Encode Profiles

How Profiles Work

Profiles let you apply different encode settings to different parts of your library automatically.

Profile priority
File queued
→
Path matches Library Profile?
→
Yes: Use Library Profile
↓
No: Use Default Profile

Default Profile

The fallback profile used for:

  • Manual encodes from the scanner (sidebar settings)
  • Webhook-triggered encodes when no library profile matches
  • Folder watch encodes when no library profile matches
  • Any setting not overridden in a library profile

Library Profiles

Per-path profiles that override the Default for specific libraries. Useful when different content needs different settings.

Example setup
/mnt/media/Movies
Archive preset (CQ 18, 40M, p7) — high quality for movies
/mnt/media/TV Shows
Stream preset (CQ 20, 25M, p6) — balanced for TV content
/mnt/media/Kids
Small preset (CQ 28, 10M, p4) — aggressive savings for kids content

Profile Settings

Each profile (Default and Library) includes all encode settings:

  • Preset, video codec, hardware (GPU/CPU), output resolution, speed
  • CQ, max bitrate, audio codec, audio bitrate, audio filter
  • DV mode, skip 4K, HDR only, discard if larger, delete original, English only
All settings are saved when you click Save in Settings. If a profile is missing a field (e.g. from an older version), it automatically inherits the Default profile's value.

Manual Encodes

When you manually scan and queue files, the settings on the left sidebar are used directly — library profiles do not override manual settings. This gives you full control when hand-picking files.

Library profiles only apply to automated encodes (Plex webhook and folder watch). Manual encodes always use the sidebar settings.

Plex Integration

Library Scanner

The scan dropdown auto-populates with your Plex media libraries. The Plex token is auto-detected from Preferences.xml — check Settings to verify token status.

Auto Library Rescan

After each successful encode, Recode automatically triggers Plex to rescan the affected library section so new files appear immediately in your Plex UI.

Plex Webhook

Plex can automatically queue new media for re-encoding when it's added to your library.

Setup
  1. Open Plex → Settings → Webhooks
  2. Add a new webhook URL:
    http://<your-server-ip>:9877/api/plex-webhook
  3. Save. New media additions will now auto-queue for encoding.
Webhook flow
New media added to Plex
→
Plex sends webhook
→
Recode matches profile
→
File queued for encode

Webhook encodes use library profiles if a match is found, otherwise the Default profile. Multi-file events (e.g. multi-episode additions) are handled — all files are individually queued.

The webhook URL is also shown in the General tab of Settings for easy copying.

Automation

Scheduled Encoding

Set a time window so new encodes only start during quiet hours. Active encodes are never interrupted — the schedule only affects when new jobs start.

Example: overnight encoding
00:00
06:00
22:00
24:00
■ Encoding allowed (22:00 - 06:00) ■ No new encodes (06:00 - 22:00)

The schedule wraps around midnight — set start=22:00 and end=06:00 to encode overnight.

Folder Watch

Monitor directories for new video files. When a new non-HEVC file appears, it's automatically queued using the matching library profile or the Default profile.

  • Paths — Add directories to watch (e.g. /mnt/media/Movies)
  • Interval — How often to check for new files (in seconds, default 300 = 5 min)
  • First scan is learning-only — existing files are catalogued but not queued
  • Only files appearing after the initial scan are auto-queued

Concurrent Encoding

Configure in Settings → General. The maximum is dynamically calculated based on your GPU VRAM. See the GPU & Concurrency tab for details.

CPU Threads

Configure in Settings → General. Controls how many CPU threads are used for CPU encoding. Set to Auto to let ffmpeg decide (uses all available cores), or set a specific number up to max CPUs minus 2. This leaves headroom for the OS and other services. Does not affect GPU encoding.

Queue Controls

The queue toolbar provides Start, Stop, and Stop & Cancel buttons. Stop prevents new jobs from starting but lets active encodes finish. Stop & Cancel also kills all running encodes immediately. Remove All clears all queued (not active) jobs.

Queue Persistence

The queue, history, and active jobs are saved to disk. On restart:

  • Queued jobs are restored
  • Interrupted encodes are re-queued at the front
  • History is preserved across restarts
  • Orphan ffmpeg processes from the previous instance are automatically killed

Service Management

  • Restart Service button available in Settings → General
  • Uses systemd path watcher (.restart-flag file) — no sudo required
  • Active encodes will be interrupted and re-queued on restart

Auto-Start Queue

By default, the queue starts paused after a server restart. You must click Start to begin encoding. Enable "Auto-start queue on server restart" in Settings → General to start automatically.

Encoded files are named dynamically: OriginalName_h265_HDR10_suffix.mkv

The tag includes codec (h265/h264), HDR type (SDR/HDR10/HLG/DV/DV-P8), and a configurable suffix. Set the suffix in Settings → General → Encode File Suffix. Only letters, numbers, hyphens, and underscores are allowed.

The interface is responsive and adapts to mobile screens. On smaller screens, the layout stacks vertically, graphs display below each other, and system stats in the header are simplified.

Shows aggregate statistics across all completed encodes — total files, space saved, average compression ratio, and total encoding time.

Auto-Updates

Plex Re-Encoder checks GitHub for new releases automatically once per day. When a new version is detected:

  • A blue banner appears below the header — click it to view the changelog and update
  • The banner persists across page reloads until dismissed or the update is applied
  • Open Help → About to check for updates and apply them
  • The full changelog is shown before you update so you can review what changed
  • Update Now downloads the release, backs up your current version, and extracts the new files
  • The service is not restarted automatically — a yellow "Service restart required" banner appears so you can restart when no encodes are running
  • Your settings, queue, history, and configuration are preserved across updates
  • A live log shows download and install progress
Backups are saved to the backups/ folder inside the app directory. You can roll back manually if needed by copying the backup files back.

Resolution Labels

All resolution displays use standard labels: 4K, 1440p, 1080p, 720p, 480p, 360p, SD. Both the source and target resolution are always shown in queue items, encoding cards, and history entries.

Installation

One-liner install from GitHub:

wget -qO- https://github.com/tarquin-code/plex-recencoder/releases/latest/download/plex-recode.tar.gz | tar xz && cd plex-recode && sudo bash install.sh

The installer includes a bundled static ffmpeg for CPU encoding and RRP binaries for remote GPU encoding — no build step needed. GPU encoding requires building ffmpeg from source via the Setup Wizard (15-30 minutes). Supported distributions: Ubuntu, Debian, Fedora, RHEL/Alma/Rocky, openSUSE/SLES, Arch/Manjaro.

About

Plex Re-Encoder is a web-based tool for batch re-encoding media libraries to HEVC (H.265) using GPU-accelerated encoding (NVENC) or CPU encoding (libx265). It integrates with Plex Media Server for library scanning, automatic webhook-triggered encoding, and post-encode library rescans.
